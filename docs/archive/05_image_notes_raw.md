> 说明：本文按阅读逻辑重排 5 张截图内容，保留原图中的标题、字段、例子、表格、代码块和关键判断。原图中已经被截断、后续不可见的地方，用“原图截断”标注。

## 0. 总体脉络

这几张图共同表达的是一个完整思路：

1. 不要一开始追求数学化的马尔可夫方程，而是先把销售交互抽成结构化样本。
2. 使用同一个基础 VLM，通过不同格式的数据分阶段训练：
   - 当前状态识别：state inference
   - 动作后果预测：transition modeling
   - 策略 / 动作偏好选择：policy preference / DPO
3. 从同一份原始轨迹或 `gt_optimal.json` 中，可以导出多种训练任务格式。
4. 实际推理时，把系统看成“agent + 内部模拟器”：先识别当前状态，再对每个候选动作分别预测后果，最后选择动作。
5. 评估时需要和通用 VLM、只做感知 SFT 的模型、完整路线模型公平比较，并做 ablation。

## 1. 训练数据和学习目标的关系

大体上是“同一个基础模型，分阶段用不同格式的数据继续训练”。

这些不同数据集，本质上可以看成是不同类型的问题-答案，只是它们教模型的能力不一样。

### 1.1 最简单的理解

假设你选一个基础模型，比如一个现成的 VLM。

然后你可以连续做几轮微调。

### 1.2 第一轮：当前状态识别

教它：

> “看这几帧图，当前顾客是什么状态？”

这时数据像普通问答：

- 问题：这几帧里用户在干嘛？要不要干预？
- 答案：用户在犹豫，主动干预分数 8，建议动作是共情安抚

这就是前面说的 SFT。

非常接近你现在已经有的 conversation/json 数据形式。

截图旁边标注：`Proactive_VLM_训练技术实战指南`

### 1.3 第二轮：动作后果预测 / 状态转移

教它：

> “如果现在采取动作 A，接下来会发生什么？”

这时数据还是问答，但问题变了：

- 问题：当前这几帧下，如果我说“选择太多确实不容易”，用户接下来会怎样？
- 答案：用户可能更防御、后退、离开，下一状态是 `E2_neg_solo`

这也是监督学习，只不过学的是状态转移，不是当前状态识别。

这一步更接近文档里说的世界模型目标：

```text
S_t + A_t -> S_{t+1}
```

截图旁边标注：`2026-0413-整体框架讨论`

### 1.4 第三轮：动作偏好 / 策略选择

教它：

> “几个动作里哪个更好？”

这时数据不再只是普通 QA，而更像“偏好比较”：

- 同一个状态下：
  - 动作 A：共情安抚
  - 动作 B：直接推荐
- 标注告诉模型：A 比 B 更好，或者 B 比 A 更差

这一步是在教模型：

- 什么时候该说
- 说哪种更合适
- 怎样少打扰用户

这就是前面说的 DPO / preference learning 那类东西。现有训练方案文档里也已经提到，第二阶段 chosen / rejected 做对齐。

截图旁边标注：`Proactive_VLM_训练技术实战指南`

### 1.5 可以写成一句话的理解

是的，就是对同一个模型，使用不同类型的数据，分阶段微调。

只是这些“不同数据集”不一定非要来自不同来源。很多时候它们可以都从同一份原始轨迹数据里转换出来。

也就是说，同一个 `gt_optimal.json`，你可以导出成三种训练集：

1. 状态理解数据集
2. 状态转移数据集
3. 动作偏好数据集

所以不是一定要三份完全不同来源的数据，而是同一批原始数据，被整理成三种不同训练任务格式。

## 2. “不同数据集”更精确的理解

“不同的数据集代表不同的问题和答案”这个理解基本对。

但需要稍微精确一点：

不是单纯“不同问题”，而是不同学习目标。

比如同样是这三帧图：

### 2.1 任务 1：当前状态识别

- 输入：三帧图
- 输出：当前状态、意图、分数、推荐动作

训练的是：感知能力。

### 2.2 任务 2：动作后果预测

- 输入：三帧图 + 动作 A
- 输出：下一状态、风险、收益

训练的是：世界模型能力。

### 2.3 任务 3：策略选择

- 输入：三帧图 + 多个候选动作
- 输出：哪个最好

训练的是：决策能力。

看起来都像“问答”，但本质上分别在训练：

- 感知能力
- 世界模型能力
- 决策能力

## 3. 数据结构：先统一成主 schema

第一步，把 Kling 生成结果统一转成一个主 schema：

- `state_id`
- `images`
- `observable_cues`
- `latent_state`
- `intent`
- `proactive_score`
- `candidate_actions`
- `best_action`
- `next_state_by_action`
- `reward_by_action`

这一步最重要。

## 4. 从主 schema 导出三套训练集

第二步，从同一主 schema 导出三套训练集：

- `state_inference.jsonl`
- `transition_modeling.jsonl`
- `policy_preference.jsonl`

这和前面的三类训练目标对应：

| 导出文件 | 对应训练目标 | 训练能力 |
|---|---|---|
| `state_inference.jsonl` | 当前状态识别 | 感知能力 |
| `transition_modeling.jsonl` | 动作后果预测 / 状态转移 | 世界模型能力 |
| `policy_preference.jsonl` | 策略选择 / 偏好对齐 | 决策能力 |

## 5. 小规模模型验证

第三步，先做一个小规模模型验证：

- 只挑 3 类 persona
- 5 类状态
- 6 类动作

这也和文档最后建议的“小规模分支式销售交互图原型”一致。

## 7. 实际使用时的推荐系统形态

我建议你把系统想成一个：

```text
agent + 一个内部模拟器
```

### 7.1 方案 A：当前最适合的版本

“同一个模型，多次调用”

这是你当前最现实、最容易先做通的方案。

## 8. 实际推理流程

### 8.1 Step 1：感知当前状态

输入：

- 当前 3 帧图
- 可选：上一轮一小段历史摘要

输出：

- `current_state`
- `intent`
- `proactive_score`
- `candidate_actions`

例如：

```text
当前状态：E3_unc_solo
当前意图：比较 / 犹豫购买
主动干预分数：8
候选动作：
```

### 8.2 Step 2：对每个动作分别做后果预测

然后系统内部循环。

对动作 A1：

输入：

- 当前 3 帧图
- 当前状态摘要
- 动作 `A1_silent`

输出：

- 下一状态
- 风险
- 收益

对动作 A2：再跑一次。

对动作 A3：再跑一次。

对动作 A7：再跑一次。

你可以得到这样的内部表：

| 动作 | 预测下一状态 | 风险 | 收益 | worth_doing |
|---|---|---|---|---|
| `A1_silent` | 继续犹豫 | 低 | 中 | yes |
| `A2_empathize` | 可能防御后退 | 中高 | 低 | no |
| `A3_recommend` | 可能觉得被推销 | 高 | 低 | no |
| `A7_disengage` | 保持舒适距离，减少打扰 | 低 | 中 | yes |

这一步正是世界模型推演。

也就是你说的“保留这些问答，然后对应不同推荐动作分别跑一下模型”。

这个思路是对的。

### 8.3 Step 3：最后再选动作

输入：

- 当前状态
- 候选动作列表
- 每个动作预测结果

输出：

- 最优动作
- 最终话术 / 是否沉默

例如：

```text
最优动作：A1_silent
原因：当前用户虽犹豫，但未显著受挫；贸然介入可能增加打扰感。短暂观察比立即推荐更优。
```

## 9. 历史记录该怎么保留

问题：

> 这是不是要“保留之前问答记录”

回答：

是，但我建议你不要原封不动塞很长聊天记录。更好的做法是保留一个结构化状态摘要。

不建议把所有历史 Q/A 全部塞进去：

- token 会越来越长
- 模型容易被旧话带偏
- 工程上不好控

## 10. 文本提取时，不要先追求方程，而要提取五元组

比起直接追求转移方程，更建议你先抽成这种格式：

```text
(persona, state, action, next_state, rationale)
```

例如：

```json
{
  "persona": "price_sensitive_cautious",
  "state": "high_hesitation",
  "action": "soft_empathy",
  "next_state": "continued_engagement",
  "rationale": "共情性话术降低被推销感，帮助用户延长停留并继续思考。"
}
```

再进阶一点可以是：

```json
{
  "persona": "price_sensitive_cautious",
  "state": "high_hesitation",
  "action": "strong_recommendation",
  "next_state_distribution": [
    {"state": "defensive", "weight": "high"},
    {"state": "leave_tendency", "weight": "medium"}
  ],
  "rationale": "强推荐可能增加心理压力，尤其对价格敏感、犹豫中的用户更容易引发防御。"
}
```

这比直接追求一个很数学化的马尔可夫方程更适合你现在的项目建设阶段。

## 11. 如何和别的 VLM 公平比较

在 salesman 场景下，建议你比较三类 baseline。

### 11.1 第一类：通用 VLM

如 Qwen2-VL / LLaVA-NeXT 之类。

看它们零样本或少样本情况下：

- 能不能看懂顾客状态
- 会不会过度干预
- 会不会只会泛泛安慰

### 11.2 第二类：你微调过的“感知版模型”

只做第一层 SFT，不做 transition / policy。

这样你能证明：

- 只做 state inference 有提升
- 但还不够

### 11.3 第三类：你完整路线模型

感知 + 转移 + 策略。

这样你能证明：

- 多轮状态转移建模带来的增益
- 策略对齐带来的增益

## 12. 重要 ablation

最重要的是做 ablation：

- 单帧 vs 多帧
- 无 reasoning text vs 有 reasoning text
- 只有 current-state label vs 加上 next-state label
- 只有 SFT vs SFT + DPO
- 无 history vs 带短历史

这样你才能回答现在最关心的问题：

> 到底是一张图够，还是多张图更好；到底只学当前状态够，还是必须学动作后果。

## 13. 整理后的执行路线

按这些截图的建议，可以整理成如下执行路线：

1. 从 Kling 或原始轨迹结果中抽取统一主 schema。
2. 主 schema 必须包含状态、图像、可观察线索、潜在状态、意图、主动分数、候选动作、最佳动作、各动作下一状态、各动作 reward。
3. 不要先建数学方程，先抽五元组：
   - `persona`
   - `state`
   - `action`
   - `next_state`
   - `rationale`
4. 从同一主 schema 导出三类训练集：
   - `state_inference.jsonl`
   - `transition_modeling.jsonl`
   - `policy_preference.jsonl`
5. 先做小规模验证：
   - 3 类 persona
   - 5 类状态
   - 6 类动作
6. 训练顺序：
   - 先 SFT state inference
   - 再 transition SFT
   - 最后 DPO / reward alignment
7. 推理时使用同一个模型多次调用：
   - 第一次识别当前状态和候选动作
   - 后续分别模拟每个动作的后果
   - 最后根据风险、收益、worth_doing 选择动作
8. 历史信息不要塞完整 Q/A，保留结构化状态摘要。
9. 对比实验要覆盖：
   - 通用 VLM
   - 只做感知 SFT 的模型
   - 完整路线模型
10. ablation 要回答：
   - 单帧还是多帧
   - 是否需要 reasoning text
   - 是否需要 next-state label
   - SFT 是否足够，还是需要 SFT + DPO
   - 是否需要短历史
